## 二、性能瓶颈分析

### 1. 主包体积过大 (1.3MB) 🔴

**问题描述：**
生产构建产物达到 1.3MB，对于 Obsidian 插件来说偏大。

**主要构成：**
- Vue 3 运行时：~300KB
- ECharts 图表库：~200-300KB（已按需加载）
- js-mdict 解析库：~100KB+
- 插件业务代码：~500KB+

**影响：**
- 插件启动时间延长
- 内存占用增加
- 首次加载体验下降

**优化建议：**
- [ ] 考虑将 ECharts 替换为更轻量的图表库（如 Chart.js、uPlot）
- [ ] 实现代码分割，按需加载统计模块
- [ ] 检查是否有未使用的依赖被打包

---

### 2. 阅读模式解析性能 🟡

**问题代码位置：** `src/views/parser.ts`

**问题描述：**
`processContent()` 方法使用多次正则表达式处理 HTML：

```typescript
// 渲染 Markdown 图片
htmlContent = htmlContent.replace(/<p>(?=.*!)(?=.*\[)(?=.*\])(?=.*\()(?=.*\)).*<\/p>/g, ...)
// 渲染多级标题（6次替换）
htmlContent = htmlContent.replace(/(<span class="stns">)# (.*?)(<\/span>)/g, ...)
// 渲染粗体（2次替换）
htmlContent = htmlContent.replace(/\*\*((?:<span[^>]*>.*?<\/span>|\s|<[^>]+>)+)\*\*/g, ...)
// 渲染斜体（2次替换）
htmlContent = htmlContent.replace(/(?<!\*)\*(?!\*)((?:<span[^>]*>.*?<\/span>|\s|<[^>]+)+)\*(?!\*)/g, ...)
// 渲染删除线
htmlContent = htmlContent.replace(/~~((?:<span[^>]*>.*?<\/span>|\s|<[^>]+)+)~~/g, ...)
```

**影响：**
- 每次翻页都重新解析整个页面的 HTML
- 长文章（>5000词）解析时可能阻塞 UI
- 正则表达式多次遍历字符串

**优化建议：**
- [ ] 缓存解析结果，相同文章不再重复解析
- [ ] 使用 Web Worker 将解析移至后台线程
- [ ] 优化正则表达式，合并处理逻辑
- [ ] 实现虚拟滚动，只渲染可见区域

### 3. 数据库查询优化 🟡

**问题代码位置：** `src/db/local_db.ts`

**问题描述：**
`getStoredWords()` 方法每次查询时重建 AC 自动机：

```typescript
async getStoredWords(payload: ArticleWords): Promise<WordsPhrase> {
    // 每次都要查询所有短语
    await this.idb.expressions.where("t").equals("PHRASE").each(...)

    // 每次都要重建 AC 自动机（O(n)复杂度）
    let ac = await createAutomaton([...storedPhrases.keys()])
    let searchedPhrases = (await ac.search(payload.article)).map(...)
}
```

**影响：**
- 短语数据量大时，构建自动机耗时
- 相同短语数据重复构建自动机

**优化建议：**
- [ ] 缓存 AC 自动机实例，短语变化时才重建
- [ ] 使用 IndexedDB 索引优化查询
- [ ] 考虑增量更新而非全量重建

### 4. Vue 组件渲染优化 🟡

**问题代码位置：** `src/views/ReadingArea.vue`

**问题描述：**
1. 使用 `v-html` 渲染大段 HTML，可能触发重排
2. 监听 `pageSize` 变化时重新计算页码和渲染
3. 单词计数监听触发频繁

```typescript
// 大段 HTML 直接渲染
<div class="text-area" v-html="renderedText" />

// 监听变化重新解析
watch([page, refreshHandle], async () => {
    renderedText.value = await plugin.parser.parse(
        article.slice(start, end).join("\n")
    )
})
```

**优化建议：**
- [ ] 考虑虚拟滚动处理长文章
- [ ] 使用 `requestIdleCallback` 延迟非关键渲染
- [ ] 为样式滑动条添加防抖（debounce）

## 三、内存使用分析

### 潜在内存泄漏风险

| 组件 | 问题 | 风险等级 |
|------|------|----------|
| `TextParser` | 每次解析创建新的 AST，单词量大时内存占用高 | 🟡 中 |
| `IndexedDB` | Dexie 实例常驻内存 | 🟢 低 |
| `Vue App` | 全局事件监听未完全清理 | 🟡 中 |
| `ECharts` | 图表实例未销毁 | 🟡 中 |

### 内存优化建议

1. **及时释放 AST** - 解析完成后立即释放中间结果
2. **销毁图表实例** - `Stat.vue` 应调用 `sevenDays.dispose()`
3. **清理事件监听** - 确保 `removeEventListener` 正确执行

## 六、总体评价

### 评分表

| 指标 | 评分 | 说明 |
|------|------|------|
| 启动速度 | ⭐⭐⭐ | 1.3MB JS 需要一定加载时间 |
| 运行流畅度 | ⭐⭐⭐⭐ | 常规使用无明显卡顿 |
| 内存占用 | ⭐⭐⭐ | 长时间使用需注意内存泄漏 |
| 大文件处理 | ⭐⭐ | 长文章解析可能阻塞 UI |
| 代码质量 | ⭐⭐⭐⭐ | 结构清晰，类型安全 |

### 总结

当前插件性能**可接受**，但在以下场景需要优化：

1. **长文章阅读**（>5000词）：解析和渲染可能耗时
2. **低配置设备**：1.3MB 主包对老旧设备有压力
3. **长期使用**：注意内存泄漏风险

### 建议实施顺序

```
Phase 1: 快速收益（1-2天）
├── 添加解析结果缓存
├── 添加 AC 自动机缓存
└── 修复 ECharts 实例销毁

Phase 2: 中期优化（1周）
├── 实现 Web Worker 解析
├── 优化正则表达式性能
└── 样式滑动条防抖

Phase 3: 长期重构（2周+）
├── 考虑 ECharts 替代方案
├── 实现虚拟滚动
└── 代码分割优化
```
